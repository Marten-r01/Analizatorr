## Core Principles

Всегда понимай задачу до начала изменений.
Всегда изучай релевантный кодовый путь до редактирования.
Всегда читай diff до того, как считать работу завершенной.
Всегда проверяй поведение релевантными проверками.
Всегда явно фиксируй допущения.
Всегда сохраняй существующие контракты, инварианты и интерфейсы, если задача явно не требует их менять.
Предпочитай минимальное изменение, которое полностью решает задачу.
Предпочитай простые решения сложным.
Предпочитай существующие абстракции новым.
Предпочитай консистентность кодовой базы личным предпочтениям.
Не смешивай несвязанные задачи в одном изменении.
Не оставляй поведение необъясненным.
Не заявляй об успехе без доказательств.

## Planning Before Changes

Перед изменением кода:

Определи точную задачу, ожидаемый результат, ограничения и то, что не входит в scope.
Определи затронутые модули, точки входа, call sites, тесты, конфиги и user-visible поведение.
Определи уровень риска:

Низкий риск: изолированное внутреннее изменение с ограниченным влиянием.
Средний риск: общий модуль, путь с чувствительностью к контрактам или stateful-поведение.
Высокий риск: auth, permissions, secrets, infra, migrations, concurrency, data integrity, networking, CI/CD, deployment, build logic, dependency management, Android runtime behavior или публичное API-поведение.

Найди существующие реализации, соседние паттерны и локальные соглашения до написания нового кода.
Определи тип изменения: bugfix, refactor, feature или cleanup. Категория должна быть явной.
Если задача крупная, разбей ее на маленькие проверяемые шаги и выполняй последовательно.
Если что-то неясно, явно зафиксируй допущение и ограничь изменение рамками этого допущения.

## Rules for Code Changes

Меняй только те файлы, которые действительно нужны для задачи.
Держи правки локальными и обратимыми.
Следуй существующим соглашениям по именованию, структуре модулей, импортам, обработке ошибок, логированию и структуре тестов.
Сохраняй публичное поведение, если задача явно не требует его менять.
Если публичное поведение меняется, зафиксируй это в итоговом отчете.
Сохраняй контракты функций и типов, если задача явно не требует их изменения.
Сохраняй обратную совместимость там, где кодовая база на нее опирается.
Переиспользуй существующие утилиты и helper-функции до создания новых.
Удаляй dead code только если это напрямую связано с задачей и безопасно.
Держи diff читаемым. Следующий инженер должен понимать намерение по одному diff.
Не делай побочных правок.
Не переименовывай файлы, символы или модули без прямой причины, связанной с задачей.
Не форматируй несвязанный код.
Не меняй комментарии, документацию или тесты, не относящиеся к задаче.
Не оставляй TODO как замену выполненной работе.

## Code Quality Rules

Пиши код, который явно выражает намерение, поддерживаем и прост для анализа.
Следуй DRY.
Следуй YAGNI.
Следуй SOLID там, где это уменьшает связанность и повышает сопровождаемость, а не создает декоративную архитектуру.
Избегай дублирования. Выноси общую логику только когда дублирование реально существует и локально подтверждено.
Не вводи преждевременные абстракции.
Не добавляй новые архитектурные слои без прямой и документированной необходимости.
Не плодись abstraction layers ради “чистой архитектуры”.
Не создавай repository, use case, interactor, reducer, wrapper или delegate просто ради формы.
Не добавляй новый паттерн, если существующий паттерн проекта уже решает задачу.
Не раздувай use cases, если они не дают реальной пользы.
Не создавай fat classes любого типа: fat Activity, fat Fragment, fat ViewModel, fat Repository, fat Mapper.
Делай control flow простым.
Делай data flow понятным.
Сохраняй обработку ошибок в рамках соглашений репозитория.
Валидируй входные данные там, где окружающий код этого ожидает.
Сохраняй типовую безопасность и консистентность интерфейсов.
Не заглушай ошибки без причины.
Не проглатывай исключения без явной логики обработки.
Не ослабляй assertions, validation или guards ради прохождения кода.
Не пиши код, который не сможешь построчно объяснить при необходимости.

## Android-specific Architecture Rules

Не выдумывай новую архитектуру, если в репозитории уже есть согласованный архитектурный паттерн.
Если архитектурный паттерн уже существует, он является обязательным источником формы решения.
Если архитектурный паттерн явно не зафиксирован, используй минимально достаточное разделение ответственности:

- UI layer отвечает за rendering, user intents, lifecycle-aware orchestration и presentation concerns.
- ViewModel или эквивалентный state holder отвечает за presentation state, event handling, coordination с domain/data и lifecycle-safe async logic.
- Domain/business logic не должна жить в composables, fragments, activities, adapters, DTO или DAO.
- Data layer отвечает за networking, persistence, parsing, serialization, caching, mapping на своем уровне и repository boundaries.
- Границы между слоями должны быть явными и предсказуемыми.

Не смешивай UI-логику, бизнес-логику и data access.
Не размещай networking, persistence, parsing, serialization, SQL, Room queries, Retrofit calls, DTO mapping или filesystem access в UI-слое.
Не обходи repository boundaries без явной причины.
Не дублируй бизнес-логику между feature-модулями.
Не создавай новые модули ради локальной задачи.
Не меняй структуру модулей без прямой причины и явного обоснования.
Не переноси код между модулями без документированной необходимости.
Предпочитай Kotlin-first решения.

## UI Layer Rules

UI-слой должен быть тонким.
Activities и Fragments должны быть thin orchestration shells, если они вообще используются в проекте.
Composables, Views, Fragments и Activities не должны содержать бизнес-логику.
UI-слой не должен напрямую дергать networking, persistence, DAO, Retrofit API, Room database, filesystem или скрытые side effects.
UI должен работать через явные state models и event flows, которые соответствуют локальному паттерну проекта.
Используй immutable UI state там, где это возможно.
Error states, loading states, empty states, permission states и content states должны моделироваться явно.
State handling должен быть явным и детерминированным.
Не кодируй UI-state через набор неявных booleans, если в проекте ожидается единая модель состояния.
Соглашения по naming, state handling, error handling и result models должны быть единообразными во всем проекте.
Не размещай mapping между DTO/domain/UI в composables, fragments или activities.
Не прячь side effects в UI callbacks без явного маршрута выполнения.
Держи composables маленькими и preview-friendly, где это применимо и поддерживается проектом.
Не делай composables stateful без необходимости.
Не используй Context там, где более уместны dependency injection, interface abstraction или platform wrapper.
Не создавай lifecycle leaks.
Не держи долгоживущие ссылки на Activity, Fragment, View, Context, Cursor, Adapter, Listener или Binder без необходимости и безопасного жизненного цикла.

## ViewModel / State Management Rules

ViewModel или эквивалентный state holder должен быть единственным источником presentation state для своего экрана, если локальная архитектура проекта не определяет иное.
ViewModel не должна быть fat class.
ViewModel не должна содержать networking implementation details, SQL, DAO details, Retrofit details или прямую работу с platform APIs без архитектурной причины.
ViewModel должна координировать зависимости через четкие интерфейсы и agreed abstractions проекта.
Не размещай тяжелую mapping logic в init block без необходимости.
Не прячь hidden side effects в init blocks.
Любой side effect в init должен быть необходим, объясним и согласован с локальным паттерном.
Не запускай неконтролируемые coroutines.
Используй lifecycle-aware coroutine scopes, согласованные со стеком проекта.
Не создавай fire-and-forget асинхронность без явной необходимости и обработки ошибок.
State transitions должны быть обозримыми и тестируемыми.
Events, actions, intents, reducers и effect handling должны следовать существующему паттерну проекта.
Не смешивай transient UI events и persistent screen state в хаотичной форме.
Если проект ожидает sealed state, sealed effect или explicit result models, агент обязан использовать именно их.
Не создавай новый стиль state management внутри одного репозитория.

## Data Layer Rules

Data layer должен быть отделен от UI.
Repositories должны быть границей доступа к данным там, где архитектура проекта этого ожидает.
Не обходи repository boundaries прямыми вызовами API или DAO из UI-слоя или state holder.
DTO, network models, Room entities и storage models не должны утекать в UI слой, если архитектура проекта ожидает явный mapping.
Mapping между DTO, domain и UI должен быть явным там, где этого ожидает проект.
Не смешивай network models и persistence models, если проект уже разделяет их.
Repository не должен становиться свалкой несвязанных источников данных.
Repository должен отвечать за согласованное поведение data access, а не за rendering logic.
DAO должен содержать только persistence concerns.
Retrofit interface должен содержать только network contract.
Mappers должны быть простыми, детерминированными и тестируемыми.
Не размещай тяжелую бизнес-логику в DTO, entity, DAO, serializer или mapper.
Error handling в data layer должен быть единообразным и соответствовать общему контракту проекта.
Caching, retry, pagination, synchronization и offline behavior нельзя добавлять стихийно. Они требуют явной необходимости и проверки влияния на контракты.

## Navigation Rules

Следуй существующей navigation architecture проекта.
Не вводи новый navigation framework без прямой необходимости и явного согласования.
Не смешивай несколько navigation approaches без причины.
Navigation destinations, route names, argument contracts и deep links должны быть единообразны.
Не хардкодь navigation strings хаотично по экрану, если проект использует centralized destinations или typed contracts.
Navigation arguments должны быть явными, минимальными и согласованными с existing destination contract.
Не передавай тяжелые объекты через navigation, если проектный паттерн ожидает ids, keys или stable arguments.
Navigation side effects должны инициироваться предсказуемо и тестируемо.
Не размещай скрытую business branching logic в navigation wiring.

## Dependency Injection Rules

Следуй единственному DI-решению, принятому в проекте.
Не подключай несколько DI-решений одновременно.
Не вводи service locator, manual singleton graph, reflection-based container или ad-hoc dependency wiring, если проект уже использует согласованный DI-подход.
Если проект использует Hilt, Dagger или другой DI stack, агент обязан следовать существующим component boundaries, scopes, modules и injection patterns.
Не меняй scopes без явной причины.
Не расширяй lifetime зависимости без необходимости.
Не инжектируй Context, если достаточно более узкой абстракции.
Не создавай global mutable singletons ради удобства.
DI configuration должна быть минимальной, читаемой и совместимой с существующим стеком.

## Gradle and Module Governance Rules

Не меняй структуру модулей без прямой причины.
Не переноси код между модулями без обоснования.
Не создавай новые модули ради локальной задачи.
Не редактируй `build.gradle`, `build.gradle.kts`, `settings.gradle`, `settings.gradle.kts`, version catalog, convention plugins, plugin management, dependency resolution strategy или buildSrc без необходимости.
Любое изменение build logic должно быть минимальным, проверяемым и документированным.
Если в проекте есть version catalog, агент обязан использовать его консистентно.
Нельзя хардкодить версии зависимостей хаотично в разных местах.
Нельзя обновлять AGP, Gradle, Kotlin, Compose, Compose Compiler, KSP, KAPT, Room plugin, Hilt plugin или test stack без проверки совместимости.
Нельзя менять namespace, applicationId, signing, flavors, build types, ProGuard/R8 rules, packaging, manifests merging behavior или test instrumentation settings вне scope задачи.
Не меняй generated sources behavior, codegen configuration или annotation processing settings без необходимости и проверки.

## Gradle and Dependency Management Rules

Добавляй зависимость только если задачу нельзя чисто решить средствами текущей кодовой базы и уже используемых стандартных инструментов.
Перед добавлением зависимости проверь, не решается ли задача уже существующей зависимостью.
Держи добавление зависимостей минимальным.
Предпочитай зрелые, узкие по назначению зависимости, согласованные с практиками репозитория.
Фиксируй причину добавления любой новой зависимости в итоговом отчете.
Не добавляй несколько зависимостей ради одной узкой задачи.
Не обновляй, не заменяй и не удаляй зависимости вне scope задачи.
Не вводи тяжелые фреймворки или новые архитектурные паттерны ради локальной проблемы.
Не добавляй транзитивную сложность без явной необходимости.

Для Android-проекта дополнительно:

Не смешивай несовместимые версии Kotlin, AGP, Gradle, Compose, Compose Compiler, Room, Hilt, Retrofit, OkHttp, Coroutines, Serialization, Navigation и test-зависимостей.
Не подключай несколько DI-решений одновременно.
Не используй несколько JSON serialization подходов без причины.
Не используй несколько networking stacks без причины.
Не добавляй дублирующие библиотеки одной категории.
Не подключай библиотеку, если аналог уже есть в проекте и покрывает задачу.
Не обновляй transitive-sensitive зависимости без оценки риска.
Не обновляй Compose Compiler, Kotlin или AGP без явной проверки совместимости друг с другом.
Не обновляй test stack так, чтобы ломалась совместимость с runtime stack.
Не обновляй версии “заодно”.
Не поднимай BOM или plugin version только потому, что вышла новая версия.
Любое изменение зависимостей должно сопровождаться:
- релевантной сборкой
- релевантными тестами
- релевантными Android-specific checks
- документированием причины
- документированием риска
- документированием совместимости

Не размещай platform-specific details в общих слоях без архитектурной причины.

Избегай lifecycle leaks.
Избегай implicit threading behavior.
State handling должен быть явным и детерминированным.

Mapping между DTO, domain и UI должен быть явным там, где архитектура проекта этого ожидает.

## Code Template and Example Rules

Codex обязан ориентироваться на корректные локальные шаблоны кода проекта.
При отсутствии шаблонов Codex обязан сначала найти соседние консистентные примеры в кодовой базе.
Новые экраны, ViewModel, repository, use case, DTO, mapper, navigation entry, UI state, tests и Gradle-конфигурации должны создаваться по единым шаблонам.
Агент не должен изобретать собственный стиль реализации, если в проекте уже есть паттерн.
Не смешивай несколько локальных стилей реализации для однотипных сущностей.
Если в репозитории есть canonical example, он имеет приоритет над личными предпочтениями агента.
Если canonical example конфликтует с новой задачей, отклонение должно быть минимальным и явно объясненным.
Новые шаблоны должны быть простыми, повторяемыми и пригодными для дальнейшего копирования.
Нельзя создавать уникальную структуру файла для одноразового кейса, если в проекте уже есть повторяемый шаблон.

## Reference Templates and Canonical Examples

В репозитории должен существовать и поддерживаться один из следующих источников истины для эталонных шаблонов:

- выделенный каталог шаблонов: [PROJECT_CANONICAL_TEMPLATES_ROOT]
- или явно индексированная секция canonical examples в [PROJECT_BUILD_AND_RUN_DOC_FILE]

Источник canonical examples должен содержать ссылки или пути на эталонные реализации для:

- feature screen
- ViewModel
- UiState
- repository
- retrofit api interface
- dto
- mapper
- room entity
- room dao
- navigation destination
- unit test
- ui test
- integration test

Если canonical example отсутствует, агент должен создать минимально консистентный шаблон, не нарушающий общий стиль репозитория.
Нельзя отклоняться от canonical examples без явной причины.
Нельзя копировать произвольный внешний стиль, если внутри репозитория уже есть собственная форма решения.

## Refactoring Rules

Делай refactor только если он напрямую помогает решить задачу или убирает локальное препятствие.
Сохраняй поведение при refactor, если задача явно не включает изменение поведения.
Отделяй refactor от bugfix или feature work, если только объединение не необходимо для безопасного выполнения задачи.
Если объединение необходимо, ограничь refactor минимумом и явно объясни, почему разделение было невозможно.
Сохраняй интерфейсы, типы, форматы данных и side effects, если задача явно не требует иного.
После refactor запускай релевантные проверки для всех затронутых путей.
Не используй refactor как предлог для широких перемещений кода.
Не меняй архитектурные границы репозитория под видом локального refactor.

## Testing and Verification

Запускай минимально достаточный набор проверок, покрывающий изменение.
Используй проектные команды, когда они доступны:

Сборка: [PROJECT_BUILD_COMMAND]
Тесты: [PROJECT_TEST_COMMAND]
Линтер: [PROJECT_LINT_COMMAND]
Проверка типов: [PROJECT_TYPECHECK_COMMAND]
Форматирование: [PROJECT_FORMAT_COMMAND]

Если в репозитории есть линтеры, форматтеры, type checks, unit tests, integration tests, ui tests, contract tests, screenshot tests или другие инструменты валидации, относящиеся к изменению, запускай их.
Для изменений бизнес-логики добавляй тесты или явно объясняй, почему тест не нужен.
Для bugfix по возможности добавляй regression test, если кодовая база это поддерживает и баг можно надежно закодировать.
Проверяй соседние сценарии, которые могли быть затронуты.
Не сообщай о завершении без фактических результатов проверок.
Если какая-либо проверка не запускалась, пиши это явно.
Если локальная верификация невозможна, точно указывай, что именно не удалось проверить и почему.
Не отключай проверки ради зеленого результата.
Не удаляй тесты ради прохождения задачи.
Не ослабляй assertions, mocks или покрытие затронутого поведения без явного обоснования, привязанного к задаче.

## Testing Rules for Android

Код должен покрываться тестами параллельно с изменениями.
Тесты должны создаваться одновременно с кодом, а не впоследствии.
В репозитории должен быть определен согласованный выделенный test root: [PROJECT_TESTS_ROOT].
Все unit, integration, UI и contract tests должны располагаться в согласованной тестовой структуре внутри выделенного test root или в явно документированном отображении этого test root на стандартные Android source sets.
Агент не должен раскладывать тесты хаотично.
Структура тестов должна быть предсказуемой и консистентной.
Тесты не должны быть вперемешку с production-описаниями и вспомогательной документацией.
Для bugfix по возможности обязателен regression test.
Для изменений UI-состояния должны быть тесты на state transitions, reducer logic или ViewModel behavior.
Для data layer должны быть тесты на mapping, parsing, repository behavior и error handling.
Для navigation и критичных пользовательских сценариев должны быть UI или integration tests, если стек проекта это поддерживает.
Если проект использует Compose, любые изменения существенной UI-логики должны получать соответствующие unit, UI или screenshot tests в рамках возможностей проекта.
Если проект использует Room, Retrofit, serialization или coroutines, измененный код должен получать релевантные tests на parsing, mapping, threading assumptions, cancellation или error propagation там, где это применимо.
Не создавай фиктивные тесты ради галочки.
Не используй overly-mocked tests, которые не проверяют реальное затронутое поведение.
Не ослабляй существующие тесты ради прохождения новой реализации.

## Debugging Rules

Воспроизводи проблему до исправления, если воспроизведение возможно.
Определи failing path, trigger condition и ожидаемое поведение.
Строй каждый шаг отладки на основе кода, логов, тестов или контролируемого воспроизведения.
Полностью читай ошибки, stack trace и failing tests до редактирования кода.
Формулируй конкретную гипотезу до внесения исправления.
Меняй по одному фактору за раз при изоляции root cause.
После исправления повторно запускай релевантный сценарий воспроизведения или тестовый путь.
Не копируй механически вывод предыдущих попыток в следующую.
Не переноси stack trace или вывод тестов в новый патч без собственного анализа.
Не делай случайных правок в надежде, что проблема исчезнет.
Не останавливайся на подавлении симптомов, если root cause можно выявить.

## Security Rules

Считай security-sensitive области зонами высокого риска.
Не логируй секреты, credentials, tokens, session data или private keys.
Не хардкодь секреты, пароли, токены, API keys, сертификаты или URL с учетными данными.
Не раскрывай внутренние секреты в коде, тестах, fixtures, документации или примерах.
Не ослабляй auth, permissions, validation, security headers, encryption, rate limits, network policy или другие защитные механизмы ради выполнения задачи.
Не меняй production config, deployment settings, CI/CD, auth flows, permission model, secret management или infrastructure settings вне явно заданного scope.
Не выполняй destructive migrations, mass update или удаление данных без явного разрешения.
Не запускай destructive shell commands без явного разрешения.

Применяй повышенную осторожность к следующим зонам:

- production configuration
- deployment manifests
- infrastructure code
- CI/CD workflows
- authentication and authorization
- permission boundaries
- network controls
- data migrations
- secrets management
- Android signing config
- release build configuration
- keystores
- Play-integrated publishing configuration
- backup/export behavior
- local data encryption
- biometric or credential flows

## Git and Change Scope Rules

Держи каждое изменение сфокусированным на одной цели.
Не смешивай bugfix, refactor и feature work в одном изменении, если это не требуется.
Если смешивание необходимо, явно объясни причину.
Держи diff минимально возможным при полном решении задачи.
Не меняй generated files, если задача явно не требует регенерации и процесс генерации доступен.
Не трогай lockfiles, если изменение зависимостей этого не требует.
Не редактируй vendored, mirrored или third-party код, если задача явно этого не требует.
Не меняй несвязанные тесты или snapshots.
Не откатывай и не перезаписывай пользовательские изменения, которые не были созданы для этой задачи.
Прочитывай весь итоговый diff перед сообщением о завершении.

## Documentation and Change History Rules

В репозитории должны существовать и поддерживаться:

- один файл истории изменений проекта: [PROJECT_CHANGELOG_FILE]
- один файл с актуальной документацией по сборке, запуску, структуре проекта и рабочим процессам: [PROJECT_BUILD_AND_RUN_DOC_FILE]

Любое значимое изменение должно отражаться в файле истории изменений.
Любые изменения сборки, setup, зависимостей, конфигурации, фичей, архитектурных решений, module boundaries, stack governance или developer workflow должны отражаться в актуальной документации.
Документация не должна расходиться с реальным состоянием проекта.
Агент обязан обновлять эти файлы в том же изменении, где меняется соответствующее поведение.
Нельзя считать задачу завершенной, если код изменен, а документация и история изменений не обновлены там, где это требуется.
Если изменение не требует обновления changelog или build/run documentation, это должно быть объяснено явно.

## Documentation Rules

Обновляй документацию, если изменение влияет на поведение, интерфейсы, конфигурацию, setup, build logic, version compatibility, dependency graph или operational expectations.
Держи документацию синхронизированной с реальным кодом.
Не выдумывай команды, переменные окружения, архитектурные детали, версии или workflow, отсутствующие в репозитории.
Используй следующие плейсхолдеры, если проектные команды неизвестны:

- [PROJECT_BUILD_COMMAND]
- [PROJECT_TEST_COMMAND]
- [PROJECT_LINT_COMMAND]
- [PROJECT_TYPECHECK_COMMAND]
- [PROJECT_FORMAT_COMMAND]
- [PROJECT_RUN_COMMAND]

Держи документацию краткой, фактической и ограниченной рамками изменения.
Не оставляй устаревшие примеры или описания поведения.
Не оставляй неверные инструкции по сборке, запуску, тестам, dependency upgrade path или module setup.

## Communication Rules

При отчете о выполненной работе:

Пиши ответы на русском языке.
Кратко описывай, что изменено.
Кратко описывай, почему это изменено.
Явно перечисляй, как изменение было проверено.
Явно отделяй факты от допущений.
Явно перечисляй риски, edge cases и непроверенные зоны.
Явно отмечай любое изменение поведения.
Явно указывай, какие проверки не запускались.
Не пиши, что задача завершена, пока не перечислены результаты проверок.
Не скрывай неопределенность.
Не выдавай допущения за факты.
Не создавай впечатление, что тесты прошли, если они не запускались.
Не заявляй уверенность выше той, которую подтверждают доказательства.

## Explicit Prohibitions

Никогда не делай следующее:

Никогда не принимай сгенерированный код вслепую.
Никогда не считай diff корректным без чтения.
Никогда не копируй stack trace, вывод тестов или текст ошибок в следующую попытку без анализа.
Никогда не делай случайные правки в надежде, что проблема исчезнет.
Никогда не оставляй код в состоянии, которое не можешь объяснить.
Никогда не добавляй зависимости без необходимости и явного обоснования.
Никогда не меняй слишком много файлов без прямой причины.
Никогда не смешивай refactor, bugfix и feature work в одном изменении без явной необходимости.
Никогда не заявляй, что задача выполнена, без проверки.
Никогда не скрывай допущения.
Никогда не выполняй destructive actions без явного разрешения.
Никогда не трогай secrets, tokens, keys, production config или infrastructure settings вне явно заданного scope.
Никогда не обходи failing tests через отключение проверок, удаление тестов или ослабление assertions, если это не было прямо запрошено и обосновано.
Никогда не пиши код, который не сможешь сопровождать и объяснить.
Никогда не выдумывай проектные команды, архитектуру, версии, стек или требования.
Никогда не делай широких переписываний там, где достаточно локального исправления.
Никогда не меняй публичное поведение молча.
Никогда не ослабляй security controls ради удобства.
Никогда не подавляй ошибки ради видимости корректности.
Никогда не форматируй несвязанные файлы.
Никогда не оставляй частично понятые изменения неотмеченными или необъясненными.
Никогда не меняй стек проекта самовольно.
Никогда не подключай альтернативную библиотеку только из личного предпочтения.
Никогда не смешивай несколько DI, navigation, networking или serialization подходов без явной необходимости.
Никогда не обновляй Kotlin, AGP, Gradle, Compose или test stack без проверки совместимости.
Никогда не размещай networking, persistence или mapping в UI-слое.
Никогда не пишите fat ViewModel.
Никогда не обновляй документацию и changelog задним числом отдельно от самого изменения, если это изменение уже требует документационных правок.

## Definition of Done

Задача считается выполненной только если выполнены все условия ниже:

Требования задачи реализованы в рамках заявленного scope.
Все затронутые кодовые пути просмотрены.
Итоговый diff полностью прочитан.
Изменение следует существующим паттернам репозитория или объясняет, почему отклонение было необходимо.
Изменение минимально, согласованно и понятно.
Релевантные тесты и проверки запущены, а результаты перечислены.
Все незапущенные проверки явно перечислены.
Все допущения явно перечислены.
Все риски и непроверенные зоны явно перечислены.
Документация обновлена, если изменилось поведение, конфигурация, способ использования, dependency graph, version contract или build/run workflow.
История изменений обновлена, если изменение значимо для проекта.
Canonical examples обновлены, если был введен новый повторяемый шаблон.
Несвязанные файлы не были изменены.
Для завершения не использовалось ни одно запрещенное действие.
